# Task Service - Решение тестового задания

Сервис для управления задачами с поддержкой периодических задач.

## Реализованная функциональность

### Типы периодичности

1. **Ежедневные задачи (daily)** - создаются каждые N дней
2. **Ежемесячные задачи (monthly)** - создаются на определенное число месяца (1-30)
3. **Задачи на конкретные даты (specific_dates)** - создаются только на указанные даты
4. **Четные/нечетные дни (even_odd)** - создаются только на четные или нечетные числа месяца

## Архитектурные решения

### 1. Модель данных

Добавлены следующие поля в таблицу `tasks`:

- `is_template` (boolean) - флаг, указывающий что задача является шаблоном для периодических задач
- `parent_task_id` (bigint, nullable) - ссылка на родительский шаблон (для созданных задач)
- `recurrence_type` (text) - тип периодичности: 'none', 'daily', 'monthly', 'specific_dates', 'even_odd'
- `recurrence_config` (jsonb) - конфигурация периодичности в формате JSON
- `next_occurrence` (date, nullable) - дата следующего создания задачи

### 2. Формат конфигурации периодичности

**Daily (ежедневные):**
```json
{
  "interval": 3
}
```
Задача создается каждые 3 дня.

**Monthly (ежемесячные):**
```json
{
  "day_of_month": 15
}
```
Задача создается 15-го числа каждого месяца.

**Specific dates (конкретные даты):**
```json
{
  "dates": ["2026-05-01", "2026-06-15", "2026-12-31"]
}
```
Задача создается только в указанные даты.

**Even/Odd (четные/нечетные):**
```json
{
  "even_odd_type": "even"
}
```
Задача создается только в четные дни месяца. Значение может быть "even" или "odd".

### 3. Scheduler (планировщик)

Реализован встроенный планировщик, который:
- Запускается как отдельная горутина при старте приложения
- Проверяет шаблоны задач каждый час (настраивается)
- Создает новые задачи на основе шаблонов, у которых `next_occurrence <= текущая_дата`
- Обновляет `next_occurrence` после создания задачи

### 4. API

Расширены существующие endpoints для поддержки периодичности:

**POST /api/v1/tasks** - создание задачи/шаблона
```json
{
  "title": "Ежедневный обзвон пациентов",
  "description": "Обзвонить всех пациентов из списка",
  "status": "new",
  "is_template": true,
  "recurrence_type": "daily",
  "recurrence_config": {
    "interval": 1
  }
}
```

**PUT /api/v1/tasks/{id}** - обновление задачи/шаблона (с пересчетом next_occurrence)

**GET /api/v1/tasks** - получение всех задач (включая шаблоны и созданные из них задачи)

**GET /api/v1/tasks/{id}** - получение задачи по ID

**DELETE /api/v1/tasks/{id}** - удаление задачи

## Принятые предположения и решения

### 1. Шаблоны vs Обычные задачи

- Задачи с `is_template = true` являются шаблонами и не выполняются напрямую
- Из шаблонов автоматически создаются обычные задачи (`is_template = false`)
- Созданные задачи имеют ссылку на родительский шаблон через `parent_task_id`
- Созданные задачи не наследуют периодичность (их `recurrence_type = 'none'`)

### 2. Обработка граничных случаев

**Месяцы с меньшим количеством дней:**
- Если указано `day_of_month = 30`, а в месяце 28/29 дней (февраль), задача создается в следующем месяце
- Это предотвращает создание задач в неправильные даты

**Високосные годы:**
- Обрабатываются автоматически стандартной библиотекой Go `time`

**Изменение/удаление шаблона:**
- При удалении шаблона созданные из него задачи сохраняются (связь через `parent_task_id` с `ON DELETE SET NULL`)
- При изменении шаблона пересчитывается `next_occurrence`, но уже созданные задачи не изменяются

**Пустой список дат в specific_dates:**
- Валидация на уровне usecase не позволяет создать шаблон с пустым массивом дат

### 3. Частота проверки шаблонов

- Scheduler проверяет шаблоны каждый час
- Это оптимальный баланс между нагрузкой на БД и своевременностью создания задач
- Можно настроить через параметр при создании scheduler

### 4. Временная зона

- Все даты хранятся и обрабатываются в UTC
- Это упрощает логику и предотвращает проблемы с переходом на летнее/зимнее время

### 5. Валидация

- `day_of_month` ограничен диапазоном 1-30 (не 31), чтобы избежать проблем с февралем
- `interval` для daily должен быть >= 1
- `even_odd_type` может быть только "even" или "odd"
- Даты в `specific_dates` должны быть в формате "YYYY-MM-DD"

## Требования

- Go `1.23+`
- Docker и Docker Compose
- PostgreSQL

## Быстрый запуск

```bash
docker compose down -v
docker compose up --build
```

Сервис будет доступен по адресу `http://localhost:8080`.

## Swagger документация

```
http://localhost:8080/swagger/
```

## Примеры использования

### Создание шаблона ежедневной задачи

```bash
curl -X POST http://localhost:8080/api/v1/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Ежедневный обзвон",
    "description": "Обзвонить пациентов",
    "status": "new",
    "is_template": true,
    "recurrence_type": "daily",
    "recurrence_config": {
      "interval": 1
    }
  }'
```

### Создание шаблона ежемесячной задачи

```bash
curl -X POST http://localhost:8080/api/v1/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Формирование отчетности",
    "description": "Создать месячный отчет",
    "status": "new",
    "is_template": true,
    "recurrence_type": "monthly",
    "recurrence_config": {
      "day_of_month": 1
    }
  }'
```

### Создание шаблона для четных дней

```bash
curl -X POST http://localhost:8080/api/v1/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Инвентаризация",
    "description": "Провести инвентаризацию склада",
    "status": "new",
    "is_template": true,
    "recurrence_type": "even_odd",
    "recurrence_config": {
      "even_odd_type": "even"
    }
  }'
```

### Получение всех задач

```bash
curl http://localhost:8080/api/v1/tasks
```
